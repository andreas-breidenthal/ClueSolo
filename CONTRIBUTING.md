#### Getting Started

1. **Fork the repository**
2. **Create a branch**
   ```bash
   git checkout -b feature/YourFeatureName
   ```
3. **Make your changes**
   - Follow existing code style
   - Comment your code
   - Test thoroughly in multiple browsers
4. **Commit with clear messages**
   ```bash
   git commit -m "Add: Description of your changes"
   ```
5. **Push to your fork**
   ```bash
   git push origin feature/YourFeatureName
   ```
6. **Open a Pull Request**

#### Code Style Guidelines

- **Use descriptive variable names**
- **Add comments for complex logic**
- **Keep functions focused and single-purpose**
- **Maintain the section organization** (use `[SECTION X]` tags)
- **Test on Chrome, Firefox, and Safari**

#### Section Organization

The code is organized into sections. When adding new code:
- Place it in the appropriate section
- If creating a new major feature, consider adding a new section
- Update the Table of Contents in the file header

### Documentation Improvements üìö

Documentation is just as important as code! Feel free to:
- Fix typos

- Add examples
- Create tutorials
- Translate to other languages

## What Should I Know?

### Project Structure

ClueSolo is a **single-file application**:
- All HTML, CSS, and JavaScript in one file
- No build process or dependencies
- Designed for maximum portability

### Key Features to Preserve

When contributing, please maintain:
- **Single-file architecture** - keep everything in one file
- **Zero dependencies** - no external libraries
- **Offline functionality** - must work without internet
- **Educational value** - maintain teaching features
- **Accessibility** - keep it usable for beginners

### Design Philosophy

- **Simplicity**: Easy to understand and use
- **Educational**: Helps teach deduction skills
- **Portable**: Works anywhere, no installation
- **Fun**: Engaging and enjoyable to play

## Pull Request Process

1. **Update documentation** if you've changed functionality
2. **Test thoroughly** across different browsers
3. **Keep changes focused** - one feature/fix per PR
4. **Describe your changes** clearly in the PR description
5. **Be responsive** to feedback and questions
